## 2. 성장과정 및 가치관

저는 새로운 기술을 배울 때 공식 문서와 표준을 코드로 검증해 원리까지 확인합니다. 네트워크 동작을 이해하기 위해 RFC 1071에 따라 ICMP 체크섬(1의 보수, end-around carry)을 구현했고, C 기반 Unix 쉘(minishell)로 파이프, 프로세스 제어, 시그널 처리를 직접 재현했습니다. 이후 어셈블리를 학습하여 레지스터 및 스택 단위를 관찰해 시스템 레벨의 동작을 확인했습니다. 이러한 학습 방식은 문제의 원인을 근본 동작에서 빠르게 좁혀 문제 해결 능력을 높였고, 프레임워크를 사용할 때도 내부 가정을 이해한 상태로 안정적인 설계를 가능하게 했습니다.

## 3. 주요 프로젝트 경험

1. ft_ping: 네트워크 진단 도구 구현

C언어와 어셈블리어를 학습하던 중 네트워크 분석 도구가 어떻게 동작하는지에 대한 근본적인 궁금증이 생겼습니다. 특히 nmap과 같은 고급 도구를 직접 구현해보고 싶다는 목표 아래, 가장 기초적인 진단 유틸리티인 ping 명령어부터 스스로 구현해보기로 했습니다. 이를 위해 ICMP 프로토콜과 RAW 소켓의 개념, 소켓 간의 정보 송수신 방식을 학습했고, 관련 내용을 RFC문서, 공식 매뉴얼 등 다양한 자료를 참고해가며 하나씩 정리해 나갔습니다. 패킷이 어떻게 구성되고 전송되는지, 또 어떻게 응답이 수신되는지를 코드로 직접 구현하며 원리를 몸으로 익혔습니다. 프로젝트를 마친 후에는 42경산의 코드 리뷰 문화에 따라 세 명의 동료에게 검토를 받아 피드백까지 반영했습니다. 예상대로 패킷 전송과 응답, RTT 측정 등 핵심 기능이 모두 정상적으로 동작했고, 이 경험을 통해 ICMP 프로토콜의 구조와 네트워크 진단 도구의 내부 작동 방식에 대한 깊이 있는 이해를 얻게 되었습니다.

2. Flutter & AWS Amplify 기반 실시간 데이팅 앱 개발

AI 기능을 활용해 사용자 얼굴 인식 및 실시간 채팅 기능을 제공하는 모바일 데이팅 애플리케이션을 개발했습니다. 초기에는 data/domain/presentation 계층으로 이루어진 클린 아키텍처 기반으로 프론트엔드를 구축하고, Firebase를 이용하여 백엔드를 구축했습니다. 하지만 프로젝트 개발 중 팀원의 “클린 아키텍처가 불필요한 복잡성을 요구하고 MVVM 모델로 충분히 개발 가능해보인다.”라는 피드백을 수용하여 MVVM 패턴으로 소프트웨어 구조를 간소화하였습니다. 그 결과, 코드 복잡도를 20% 이상 절감하고 구조 개선 후, UI 관련 버그가 빠르게 탐지되는 등, 버그 수정 소요 시간도 단축되었습니다. 또한 팀원간의 회의에서 Firebase의 AI 기능의 한계로 인해, AWS의 AI를 이용하자는 의견을 받아들여 AWS Amplify를 통해 AppSync GraphQL API와 Lambda 함수를 중심으로 서버를 구축하고, WebSocket 기반 실시간 통신으로 메시지 전송·구독 기능을 구현했습니다. 이 경험을 통해 초기 설계를 맹목적으로 따르기보다, 실제 프로젝트 요구와 팀의 의견을 반영해 기술과 아키텍처를 유연하게 개선하는 능력을 갖추게 되었습니다.

3. minishell: Unix 쉘 구현 (C)

유닉스 쉘의 핵심 기능을 C언어로 직접 구현한 프로젝트입니다. 파이프와 리다이렉션을 통한 프로세스 간 통신, 환경 변수 관리, 시그널 처리 등을 구현하며 운영체제의 프로세스 관리와 IPC(Inter-Process Communication) 메커니즘을 체득했습니다. 이 과정에서 시스템 콜 문서를 참고하며 fork, exec, wait 등의 프로세스 제어 함수를 깊이 이해했고, 메모리 관리와 에러 처리의 중요성을 배웠습니다.

## 4. 성장 포인트: 문제 해결 경험

Flutter로 카메라 기능 구현 중 전·후면 전환 시 미리보기 화면이 검은색으로 렌더링되는 문제에 직면했습니다. 당시 Riverpod 기반 상태 관리를 사용 중이었는데, 상태 변화 흐름이 복잡해 디버깅에 2일 이상 소요되었습니다. 팀 회의를 통해 근본 원인을 분석한 결과, 불필요한 상태 중복과 라이브러리의 복잡성이 문제임을 인식하고, Riverpod를 제거하고 ChangeNotifier 기반 MVVM 패턴으로 재설계하기로 결정했습니다. 또한, ViewModel에 sealed class로 카메라 상태(Preparing/Ready)를 명확히 정의하고, async/await로 미리보기 초기화 완료 시점에만 Ready 상태로 전이하도록 개선했습니다. 간단 로깅으로 흐름을 추적한 결과 크래시 0건, 지연 약 0.5ms이슈만 남아 Github Issue로 관리 중이며, 프로필 사진 촬영 플로우 개발을 맡은 팀원으로부터 재발 보고가 없었습니다. 이 경험을 통해 때로는 최고 기술이 아닌 문제에 맞는 단순한 구조가 최고의 해결책임을 배웠고, 기술 선택 시 팀의 디버깅 가능성과 유지보수성을 함께 고려하는 판단 능력을 갖추게 되었습니다.

이전 프로젝트에서 백엔드 API 구현을 담당했고, 프론트엔드 파트를 이어 받아 완성하는 과정에서, 마지막 테스트 단계에서 변수 네이밍 불일치(백엔드는 카멜케이스, 프론트엔드는 스네이크케이스)를 발견하는 데 2시간이 소요된 경험이 있습니다. 이 시행착오를 계기로, 다음 프로젝트에서는 초기에 팀 전원이 참여해 네이밍 규칙 등 범용적 개발 표준을 명확히 수립하고 진행하여, 불필요한 오류 및 커뮤니케이션 비용을 효과적으로 줄일 수 있었습니다. 이러한 경험을 통해, 개발 과정에서 작은 규칙 하나가 생산성과 결과물 품질에 큰 영향을 미칠 수 있음을 깨닫고, 체계적인 팀 규칙 설정을 습관화하게 되었습니다.

C언어로 직접 메모리 관리 오류를 추적·해결하며 포인터와 힙 구조의 동작 원리를 깊이 이해했습니다. 또한 어셈블리어로 디버깅 루틴을 작성해 레지스터 값과 스택 프레임을 분석하며, 시스템이 명령 단위로 동작하는 과정을 체득했습니다. 이러한 경험을 통해 단순 코드 수정이 아닌, 문제의 근본 원인을 파악하고 시스템 수준에서 해결책을 설계하는 역량을 갖추게 되었습니다.

## 5. 입사 후 포부

입사 후 첫 3개월은 회사의 코드베이스와 개발 문화에 빠르게 적응하며, 기존 시스템의 아키텍처를 깊이 이해하는 데 집중하겠습니다. 42경산에서 배운 동료 학습(Peer Learning) 문화를 바탕으로 팀 내 코드 리뷰에 적극적으로 참여하며, 선배 개발자들의 피드백을 통해 실무 역량을 빠르게 성장시키겠습니다.

6개월 차부터는 담당 서비스의 성능 최적화나 신규 기능 개발에 주도적으로 기여하고, 시스템 프로그래밍과 네트워크에 대한 저수준 이해를 바탕으로 복잡한 기술적 문제를 해결하는 역할을 수행하고자 합니다. 특히 백엔드 서버의 병목 지점 분석, 데이터베이스 쿼리 최적화, 인프라 자동화 개선 등 시스템 전반의 안정성과 효율성을 높이는 작업에 기여하겠습니다.

장기적으로는 풀스택 개발자로서 프론트엔드부터 백엔드, 인프라까지 아우르는 시스템 아키텍처를 설계하고, 기술 블로그 작성이나 사내 세미나를 통해 팀의 기술 역량 향상에도 이바지하고 싶습니다. 제가 42경산 에서 경험한 지식 공유와 협업의 가치를 회사에서도 실천하며, '같이 일하고 싶은 개발자'로 성장하겠습니다.
